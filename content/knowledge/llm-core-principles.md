# LLM核心原理

## 1、Attention（注意力机制）

计算当前词对其他词的关注程度。一个句子进入Attention阶段，根据Embedding计算每个词的Q、K、V。这三个分别是Q：我想要找到什么信息？K：我能够提供什么特征？V：如果你关注我，我把什么信息给你？Attention不是找到最相关的词，而是根据所有词的重要程度，对所有词的信息进行加权融合，从而得到当前Token新的表示。

**Self-Attention（自注意力）**：让模型在理解一个词时，会自动去寻找句子中其他词和它的关联度。可以理解成模型在理解上下文，不然使用指代词（比如它、这样）的词模型不能理解其含义。

**Multi-Head Attention（多头注意力）**：将注意力机制分成多批次同时进行。对一个词理解是，从对各角度去分析（比如词义，上下文关联等），解决了理解不全面的问题。

**Position Encoding（位置编码）**：Transformer是将一整句话塞进模型，所以不做特殊处理则不知道每个词的位置，Positon Encoding将词和编码做了一个映射，也就是位置编码，这样就能够知道每个词是在第几个位置了。

**FFN（前馈神经网络）**：将每个词升维，更加细致的了解这个词的语义。确保理解正确后再进行降维。

## 2、Residual（残差连接）

保留上一层输入的信息，并学习需要补充的东西。Residual的重要作用体现在将当前模块的输入token的语义融合自己的原始语义，形成一个新的、更加丰富的表示。

准确的表达：Residual将当前模块的输出与该模块的输入相加，从而保留已有的信息，同时加入新的信息。

Attention 负责信息交互，Residual 负责信息保留，FFN 负责语义加工。

## 3、LayerNorm（层归一化）

不负责理解语义，它负责让每一层输出保持稳定，从而让 Transformer 能够持续、稳定地训练。LayerNorm 调整的是数值分布，而不是语义本身。

准确的表达：LayerNorm（层归一化）的作用是将每个 Token 的表示调整到稳定的数值分布，使不同模块输出的表示保持一致的数值尺度，从而让后续模块能够稳定地继续学习，而不会改变 Token 本身表达的语义。

## 4、Transformer Block（最重要）

 Token 交流（Attention）、保留已有信息（Residual）、稳定数值分布（LayerNorm）、深入加工语义（FFN），最终得到越来越符合上下文的表示。这一个过程，我们就可以称之为一个Transformer Block。

## 5、Encoder 和 Decoder

**Encoder**：负责理解，它使用了**Embedding、Position Encoding、多层Self-Attention**。它的输出是语义理解向量，不再是词语

**Decoder**：负责根据理解（向量）生成新的词语，通过"逐 token 生成"的方式不断预测下一个词。

**Encoder：**把输入语言编码为包含上下文信息的语义表示（Context Representation）；**Decoder：**基于这些语义表示和已生成的 Token，自回归地预测下一个 Token；最后由 Tokenizer 将 Token ID 解码为自然语言文本。

## 6、GPT到底干了什么？

根据已有的上下文，不断**循环**预测下一个最可能出现的token，并把最有可能的token加入上下文来去预测下一个token，直到生成最终的回答。

## 7、LLM是如何训练

**训练在干什么？**大模型是根据海量的文本不断预测、纠错和调整参数，把语言、事实和模式中的统计规律编码到参数中。这样，当它遇到新的问题的时候能够根据已有的规律进行作答，而不是传统的在数据库中查找答案。

**训练数据从哪里来？**

LLM 的训练数据并不是来自某一个地方，而是综合了公开互联网内容、公开出版物、授权数据、人工标注数据以及越来越多的高质量 AI 合成数据。经过严格的清洗、筛选和处理后，这些文本才会成为模型的训练语料，帮助模型学习语言规律、知识模式和推理能力。

7.1 公开的互联网文章

互联网公开的文本，例如技术博客（CSDN）、新闻网站、开源文档、开发者论坛、维基百科、stackOverflow等。

7.2 公开的出版物

学术论文、教育资料、政府公开的文件、法律法规等

7.3 授权数据

由大模型企业与其它公司合作获得公司授权从而获得公司内部数据

7.4 人工标注的数据

7.5 AI生成的数据

利用已有的大模型自动生成高质量问答、代码、数学题、推理样本等，再经过筛选后继续用于训练，可以有效提升模型在特定能力上的表现。

**训练的本质**：不断调整模型参数，使得模型对下一个token的预测越来越准确。训练的本质不是把知识"塞进"模型，也不是让模型背答案，而是通过"预测 → 计算误差 → 调整参数 → 再预测"的循环，让数十亿甚至数千亿个参数逐渐形成对语言、知识和规律的内部表示。最终，模型能够根据上下文生成最合理的下一个 Token，这种能力进一步表现为我们看到的写作、编程、翻译和推理等能力。模型最终学到的不是具体答案，而是语言和世界中的统计规律，因此它能够泛化到训练数据中没有出现过的新问题。

**训练过程：**

LLM的训练分为预训练和后训练两个阶段

**预训练**的核心是自监督学习，基于海量的文本数据，采用Next Token Prediction方法，即给定前文，预测下一个token。通过Cross-Entropy Loss计算误差，再反向传播更新模型参数（可能是数十亿至万亿参数）。这个过程是让模型把语言的统计规律、世界知识、逻辑模式都压缩进参数里。

> Loss（损失函数）用于衡量模型预测结果与真实答案之间的差距，Loss 越小说明模型预测越准确。

**后训练**指使用了SFT（监督微调），用高质量人工标注的指令-回答数据，让模型学会遵循指令。同时基于**人类反馈的对齐（RLHF）**，使用强化学习算法（例如PPO）让模型的输出更符合人类偏好。

> **RLHF（Reinforcement Learning from Human Feedback）：**人类反馈的对齐

## 8、推理流程和优化

**8.1 推理流程**

推理（Inference）：利用训练阶段所学到的知识，根据用户输入进行计算，然后生成下一个Token

流程是用户输入，然后使用Tokenization进行分词，然后计算每个词的向量，给每个词进行位置编码，Transformer得到下一个token的概率，Sampling（？），输出第一个token，然后使用Decode将token还原为语言。持续这个过程，直到结束。

**8.2 Tokenization**

模型不能够直接理解文字，而是调用Tokenizer将一个句子进行分词，并给每个字一个Token ID

**8.3 Embedding**

将token语义向量化，将每个Token ID都映射为高纬度向量，最终将所有语句进行语义向量化。

**8.4 Prefill**

Transformer开始计算，首先是将用户的Prompt完整阅读，并计算一遍（Attention），这一步即是完成了Prompt上下文的理解（将所有token的语义向量都计算出来），同时会把后续生成所需要的KV Cache保存好。

Transformer的输出不是一句话，而是一个Token概率表上的得分（Logits），然后使用Sampling将得分转换为概率，也就是说，Transformer的回答不是一整句话，而是下一个Token最有可能是什么。

**8.5 Decode**

Decode 是推理阶段的逐 Token 生成过程，也是 LLM 自回归（Autoregressive）生成的核心阶段。模型会将上下文（已生成的 Token 与原始 Prompt） 一起作为新的输入，在 Prefill阶段生成的KV Cache 的帮助下逐轮预测下一个 Token，并重复这一过程，直到生成结束标记（EOS）。需要注意，这里的 Decode 指的是推理阶段（Generation Stage），而不是 Tokenizer的decode() 操作。真正将 Token ID 转换为自然语言文本的是 Tokenizer的decode()操作。

**8.6 Sampling**

根据Transformer输出的Logits（通常会先转换为概率分布，并结合不同策略），选择最终输出Token的过程。

常见的Sampling策略：（这些在部署Deepseek时，编写modelfile有涉及过）

- Temperature
    
    调整概率分布的陡峭程度，若Temperature越低，概率分布越陡峭，则Token概率大的被选中的概率越大；若Temperature越高，则Token概率将会更平均，Token概率低的被选中的几率也会增加。
    
- Top-K
    
    名称与RAG中的Top-K相同，但作用对象不同。RAG的Top-K是选择相关文档，而 Sampling 的 Top-K 是选择候选 Token。
    
- Top-P
    
    固定累计概率，按照 Token 概率从高到低排序，保留累计概率达到P的最小候选集合，再从该集合中进行采样。
    
- Greedy Search
    
    不属于随机采样，而是始终选择概率最高的Token，因此生成结果最稳定，但容易缺乏多样性。
    

Sampling 并不会改变 Transformer 对下一个 Token 的预测结果，而是在预测结果的基础上，按照不同策略选择最终输出的 Token。不同的 Sampling 策略会影响模型生成内容的随机性、多样性和稳定性。

**8.7 推理优化**

8.7.1 KV Cache

如果没有KV Cache，每生成一个Token，都需要重新计算上下文，导致大量重复的计算。根据之前对于Attention机制的了解，只有新的Token需要有Query，且Query只作用于新的Token，而历史Token的Key和Value会被后续所有Token反复使用，所以需要将他们缓存。

8.7.2 Prefix Cache

将多个请求中相同前缀（Prefix）对应的KV Cache保存下来，当新的请求拥有相同前缀时，直接复用这部分KV Cache，而不需再次执行Prefill，从而进一步降低推理延迟，提高 GPU 利用率。。

> 举例：假设用户A的提问是：你是一个Python老师，请解释一下for循环。用户B的提问是：你是一个Python老师，请解释一下什么是构造器。那在这个例子中，你是一个Python老师这句话只需要一次计算存入KV Cache，用户B提问时直接调用即可，不需要去做重复的计算。

在上面这个例子中，Prefix就是"你是一个Python老师"

Prefix Cache与KV Cache的区别在于一个是在同一个请求中反复使用，一个是在不同请求中反复使用。它有一个局限性，就是必须一摸一样的文案才会触发Prefix Cache，比如"你是一个Python老师"，这个Prefix必须每个字都相同才能在不同的请求中使用。

8.7.3 Continuous Batching

一种动态批处理技术。

假设一个GPU一次可以处理四个请求，但是第二个请求耗时10min，其余的耗时3min，那这就会导致其他三个请求结束了，但是第二个请求还在执行，需要等待执行完成后才能接收新的四个请求。现在使用Continuous Batching，GPU不需要等待整个Batch结束，而是只要有请求结束了，就立刻加入新的请求进入Batch开始计算，从而保持GPU的高利用率。

KV Cache与Continuous Batching高度相关联，因为每个请求都维护了独立的KV Cache。

8.7.4 Flash Attention

针对Attention计算过程的优化算法，它的主要作用是通过减少GPU显存与计算单元之间的数据搬运，提高计算速度并且降低显存占用。它通过分块（Tiling）计算，并充分利用 GPU 的高速共享内存（Shared Memory），减少了显存（HBM）的数据读写次数，在不改变模型输出结果的前提下，提高了 Attention 的计算速度，并降低了显存占用，尤其适用于长上下文场景。

8.7.5 Paged Attention

（可以类比动态数组和固定长度的数组）传统KV Cache连续分配，导致大量显存空间被浪费，使用Paged Attention使得GPU能够同时容纳更多的请求，提高推理的容量，传统的方式是在一开始就申请固定长度的显存，而使用了Paged Attention，它借鉴了操作系统分页（Paging）的思想，将 KV Cache 拆分为多个固定大小的数据块（Page）进行管理。

8.7.6 Quantization（量化）

目前市面上的LLM，会提及一个参数多少，比如Deepseek-V1-70B，这里的70B就是指大模型的参数，也就是70亿个数字，这些参数的表示又可以分为：FP16、FP32，FP16代表每个参数占用16bit，常见的精度有：

| 精度 | 每个参数占用 | 特点 |
| --- | --- | --- |
| FP32 | 32 bit | 精度最高，体积最大 |
| FP16 | 16 bit | 推理最常见 |
| BF16 | 16 bit | 与 FP16 类似，但数值范围更大，训练中更稳定 |
| INT8 | 8 bit | 模型更小，速度更快 |
| INT4 | 4 bit | 极致压缩，本地部署常见 |

量化的概念是使用更低精度的数据类型来存储和计算模型的参数，从而减少模型占用存储空间和显存，并提升推理速度。

为什么降低了参数精度对于模型来说影响不大，假设某个参数是0.728452，现在给它降低精度，它变成了0.7285，它占用的空间变少了，但是在70亿参数的大模型中，这点变化对于整体的影响非常有限。好比一张4000万像素的图片，压缩一点，肉眼是无法看出差别的。

当然，精度下降，势必会导致答案的质量出现下降，量化程度越高，则误差越大。

这里我们暂且不说量化的一些常见方案。

量化解决的一个最主要的问题是能够使用更少的空间来部署一个相对较强的大模型。

8.7.7 Speculative Decoding（推测解码）

Token逐个生成，每次都需要Transformer计算，导致生成的速度必然会变慢。于是一种新的想法：先使用小模型预测多个Token，再由大模型一次性验证（并行，快的原因也在这里）这些预测是否正确的推理优化技术，从而减少大模型的推理次数，提高生成速度。

> 假设老师作业是学生写一句改一句，这样会很慢。现在变成学生一次性写好十句，然后老师快速检查，如果前八句都是正确的，那就从第九句开始重新写。

推理优化的本质，都是围绕三个点：1、减少重复计算；2、提高计算效率；3、减少资源消耗。
